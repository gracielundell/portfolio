Angular 2.0 vs Ember

April 3, 2016

Currently I am enrolled in the Portland-based code school, Epicodus. Last week, we learned the basics of the new JavaScript framework, Angular 2.0. Angular 2.0 is a framework that uses Typescript to compile you written code into JavaScript to be implemented on the web. It uses a local server to update and run projects through the Node Project Manager (npm) toolkit, Gulp. The server is run by first running $ gulp build, followed by $ gulp serve.

This week, we are learning another JavaScript framework, Ember. Ember is a opinionated development system, which means that best practices are not only encouraged, but must be used to get the project to run on the server. The server is hosted locally and uses $ ember s to activate.

This post will be about my experience using the two frameworks, and a beginner’s opinions about the two.

Let’s take a closer look at each:

Angular 2.0

Something that I like very much about Angular 2.0 (I have no experience with Angular 1) is that you can clearly tell what files the one you are working on is interacting with. At the top of every file is a list of what files must be imported to run the code on that page:

            import { Component, EventEmitter } from ‘angular2/core’;
            import { MealListComponent } from ‘.meal-list.component’;
            import { Meal } from ‘./meal.model’;

For example, here is the beginning of my parent file for a meal tracking app. This file is the base of the program, getting information from the meal list, the model (Meal), and Angular’s own framework to create the emitter and components. This gives the programmer a clear idea of what the functionality of this page of code is trying to/will do. It makes it easy for someone who may not be familiar with the code to follow its routing between components.

Not only can you see where you are sending and retrieving data from, all of your working files reside inside the same folder, App. This makes it easy to move between pages and keep track of where information is going.

I also very much enjoy the way Angular is compiled: the programmer only ever writes in Typescript; the framework takes care of all the compiling for you in a behind-the-scenes fashion. The project runs on your local server (localhost:3000), but does not need a database. This way, you only need to worry about one language’s syntax/errors/etc. The HTML components may be written on the same file as the Typescript that controls its functionality, or may be moved to a separate file for more modularity.

Ember

Ember is an opinionated language, meaning that it has set rules for how files may communicate and is more picky about syntax than a non-opinionated one. The pro of this is that every project in Ember must follow a similar path to completion, and switching between projects is made easier because you know where everything is. The downside, and this may just be an Ember thing, is that there ARE SO MANY FILES AND FOLDERS. Everything is completely modular too, with JavaScript elements residing in .js files (and multiple folders that contain .js files); while Handlebars elements reside in .hbs files (Handlebars is essentially written with HTML elements, but uses {{ }} to pull in (usually) user interface data through JS).

I feel like currently I waste so much time just trying to find what file it is I need to write something in, and follow where it’s sending its information to. Though, with time I think this will become one of the things to love about Ember: once you figure out the file path, the code is so modular and organized. It also enforces that you actually grasp how the files relate. It’s up to you to know the path the data needs to be sent along to get from the children files to their parents and the model.

Another thing that is great about Ember is that it’s super command-line friendly, with most files being created there. To begin a project, one types $ ember generate new [file-name] into the location they want the project to reside in their terminal. Then $ cd into this file and begin working. Your README.md, package.json, all npm and bower components, as well as app, styles, config, components, tests, template, etc files are there for you already. What a dream! Adding new components and models are also generated through the command line and will create all the necessary files for you with their proper exporting code.

Something I don’t really like about Ember is that you’re writing in two languages on one project: JS and HBS. Though the more time I spend with this framework the less this bothers me. The .hbs files are all dealing with and taking in/spitting out UI, while the .js files are performing the actual functionality of the code. In JavaScript. When working in Angular 2.0 and Typescript, I felt it was hard to retain what was language, Typescript and what was framework. I feel in Ember the two are more separated and therefore more identifiable.

To Conclude...

I am someone who learns best if they can identify a pattern and follow it through different circumstances. Ember is full of patterns: where the data goes, how it jumps from a .hbs file, into its corresponding .js file, back to a .hbs, and then a final jump into the .js model. I feel like though there may be a few more intuitive things in Angular 2.0, I want to understand Ember more. I like the idea of being able to jump into another’s code and know where everything should be. I feel like after a week I am finally understanding where and how the paths access and pass information from file to file. It’s a struggle, but one I find myself wanting to win for the feeling of accomplishment that I know will come with it.
